# 第 1 章：UNIX 基础知识

## 1.1 引言

APUE 开篇就说：**本书描述的是 UNIX 系统的程序设计接口**。

你平时用 shell 命令，比如 `ls`、`cat`、`grep`，这些命令背后都是调用了操作系统提供的**系统调用（system call）** 和 **库函数**。这本书就是教你直接用这些接口写程序，而不是只用命令行工具。

库函数不能直接操作内核，只能通过系统调用。 
调用链是单向的： 库函数 → 系统调用 → 内核 → 硬件

类比：
- 你现在用的是"餐厅"（shell 命令）
- APUE 教你进"厨房"（直接调用系统接口）

---

## 1.2 UNIX 体系结构

```
┌─────────────────────────────┐
│         应用程序              │  ← 你写的程序、shell 命令
├─────────────────────────────┤
│    shell / 库函数 (libc)     │  ← printf、fopen 等
├─────────────────────────────┤
│      系统调用接口             │  ← write、open、fork 等
├─────────────────────────────┤
│         内核 (Kernel)        │  ← 管理硬件、内存、进程
├─────────────────────────────┤
│           硬件               │
└─────────────────────────────┘
```

关键区别：
- **系统调用**：内核提供的接口，数量有限（Linux 约 300 个），直接陷入内核执行
- **库函数**：基于系统调用封装的，更方便使用，比如 `printf` 内部调用了 `write`

你写 Python 时用的 `open()`，底层就是调用了 C 库的 `fopen()`，再往下是系统调用 `open()`。

## 1.3 登录

UNIX 系统启动后，内核运行的第一个用户进程是 `init`（现在是 `systemd`），它负责启动登录程序。

登录时系统做了什么：
1. 读 `/etc/passwd` 找你的用户信息
2. 验证密码
3. 启动你的 **shell**（登录 shell）

`/etc/passwd` 每行格式：
```
maverick:x:501:501::/home/maverick:/bin/zsh
  用户名  密码 UID GID  主目录      登录shell
```

密码字段是 `x` 表示实际密码存在 `/etc/shadow`（加密存储）。

**UID/GID** 是 UNIX 权限系统的基础，文件权限章节会深入讲。

## 1.4 文件和目录

**一切皆文件** — UNIX 的哲学。普通文件、目录、设备、网络套接字，在内核里都用统一的文件接口操作。

**目录结构：**
```
/
├── bin/      # 基本命令
├── etc/      # 配置文件
├── home/     # 用户目录
├── dev/      # 设备文件
├── proc/     # 进程信息（虚拟文件系统）
└── usr/
    └── lib/  # 库文件
```

**路径：**
- 绝对路径：从 `/` 开始，如 `/home/maverick/file.txt`
- 相对路径：从当前目录开始，如 `../file.txt`

“创建新目录时会自动创建了两个文件名：.（称为点）和..（称为点点）。点指向当前目录，点点指向父目录。在最高层次的根目录中，点点与点相同。”

每个进程都有一个**工作目录（current working directory）**，这就是你在 shell 里 `cd` 切换的那个目录。

## 1.5 输入和输出

### 文件描述符（File Descriptor，fd）

每个打开的文件在进程里用一个**非负整数**表示，这就是文件描述符。

三个默认已打开的 fd：

| fd | 名称 | 对应 |
|----|------|------|
| 0 | 标准输入 stdin | 键盘 |
| 1 | 标准输出 stdout | 终端 |
| 2 | 标准错误 stderr | 终端 |

你在 shell 里用 `>` 重定向，本质就是把 fd 1 指向文件。

### 第一个程序（cat 的最简实现）

```c
#include "apue.h"
#include <unistd.h>

#define BUFFSIZE 4096

int main(void)
{
    int     n;
    char    buf[BUFFSIZE];

    // 从 fd 0 (stdin) 读，写到 fd 1 (stdout)
    while ((n = read(STDIN_FILENO, buf, BUFFSIZE)) > 0)
        if (write(STDOUT_FILENO, buf, n) != n)
            err_sys("write error");

    if (n < 0)
        err_sys("read error");

    exit(0);
}
```

注意：
- `read` / `write` 是**系统调用**，直接操作 fd
- `STDIN_FILENO` = 0，`STDOUT_FILENO` = 1，定义在 `<unistd.h>`
- `err_sys` 是书里封装的错误处理函数，打印错误并退出

## 1.6 程序和进程

**程序 vs 进程：**
- **程序**：磁盘上的可执行文件（静态的）
- **进程**：程序运行起来后在内存中的实例（动态的）

每个进程有唯一的 **PID（进程 ID）**，你用 `ps` 命令看到的数字就是 PID。

```c
#include "apue.h"

int main(void)
{
    printf("hello world from process ID %ld\n", (long)getpid());
    exit(0);
    // 每次运行 PID 不同
}
```

**进程控制的三个核心系统调用**（后面会深入讲）：
- `fork()` — 创建子进程
- `exec()` — 执行新程序
- `waitpid()` — 等待子进程结束

shell 执行每条命令的方式：`fork` 出子进程，子进程调用 `exec` 执行那个命令，父进程（shell）调用 `waitpid` 等待子进程结束。

## 1.7 信号

**信号（Signal）** 是通知进程发生了某个事件的机制。

`Ctrl+C` 终止程序，就是发送了 `SIGINT` 信号给前台进程。

```c
#include "apue.h"

static void sig_int(int signo)  // 信号处理函数
{
    printf("interrupt\n");
    // 不退出，继续运行
}

int main(void)
{
    signal(SIGINT, sig_int);  // 注册信号处理函数
    // 按 Ctrl+C 不会退出，而是打印 "interrupt"
}
```

信号处理三种选择：
1. **默认动作**（通常是终止进程）
2. **忽略信号**
3. **自定义处理函数**（如上）

## 1.8 时间值

UNIX 里两种时间要区分：

- **日历时间（Calendar Time）**：从 1970-01-01 00:00:00 UTC 至今的秒数，类型 `time_t`，即 Unix 时间戳
- **进程时间（Process Time）**：进程消耗的 CPU 时间，用 clock ticks 计量

进程时间分三类：
- **时钟时间（real）**：从开始到结束的墙钟时间
- **用户 CPU 时间（user）**：执行用户代码的时间
- **系统 CPU 时间（sys）**：执行内核代码的时间（系统调用里耗费的）

> 你用 `time python train.py` 时看到的 `real/user/sys` 就是这三个。

## 1.9 系统调用和库函数

| | 系统调用 | 库函数 |
|--|---------|--------|
| 实现位置 | 内核 | 用户空间 |
| 举例 | `write()` | `printf()` |
| 是否陷入内核 | 是 | 不一定 |
| 性能开销 | 相对大 | 相对小 |

`printf` 有内部缓冲区，攒够数据再调用 `write`，所以比直接调用 `write` 效率高（减少系统调用次数）。

## 延伸问题与补充

### 库函数能直接操作内核吗？

不能。库函数运行在用户态，访问内核的唯一合法入口是系统调用。调用链是单向的：

```
库函数 → 系统调用 → 内核 → 硬件
```

CPU 有两种运行模式：
- **用户态（User Mode）**：普通程序运行，不能直接访问硬件和内核内存
- **内核态（Kernel Mode）**：内核运行，可以做任何事

每次系统调用都会触发 CPU 从用户态切换到内核态，执行完再切回来，这个切换有性能开销，这也是标准 I/O 加缓冲区的意义——减少系统调用次数。

---

### shell 是用系统调用还是库函数写的？

两者都用。shell 是普通的用户态程序，该用库函数用库函数，该用系统调用用系统调用。

```
shell 执行 ls 命令
├── fork()          ← 系统调用，创建子进程
├── execve()        ← 系统调用，加载 ls 程序
├── waitpid()       ← 系统调用，等待子进程结束
├── printf()        ← 库函数，打印 prompt
└── readline()      ← 库函数，读取用户输入
```

`/bin/ls` 是磁盘上独立的可执行文件，shell 的工作是：解析输入 → 找到可执行文件 → fork+exec 运行它 → 等它结束。

shell 更像**命令调度员**，而不是解释器。shell 脚本（.sh）才是被 shell 解释执行的，那个场景下 shell 才类似 Python 解释器。

### POSIX、GNU、Linux、macOS 的关系

- **POSIX**：IEEE 制定的标准，规定了系统调用有哪些、库函数有哪些、shell 行为等。GNU 工具集面向 POSIX 标准写，不依赖具体内核。
- **GNU**：1983年 Stallman 发起，写了 bash/gcc/ls/glibc 等工具集，但内核（Hurd）难产。
- **Linux**：1991年 Linus 写的内核，和 GNU 工具集合并成 GNU/Linux。
- **macOS**：内核是 XNU（Mach 微内核 + BSD），通过了 POSIX 官方认证，是正牌 UNIX。Linux 兼容 POSIX 但没有认证。

```
UNIX
├── BSD → FreeBSD → XNU → macOS
└── 商业 UNIX（AIX、Solaris）

POSIX 标准
├── GNU 工具集 + Linux 内核 = GNU/Linux
└── XNU 内核 = macOS
```

验证 macOS 内核：
```bash
uname -a        # 输出 Darwin，XNU 的操作系统名
ls --version    # BSD 版本会报错，GNU 版本会打印版本信息
```

### 用户态和内核态切换的原理

CPU 有特权级寄存器（x86 叫 Ring），Ring 0 是内核态，Ring 3 是用户态。用户态下 CPU 电路会直接拦截访问硬件的指令，不是软件检查，绕不过去。

切换过程：
```
用户态执行 syscall 指令
    ↓
CPU 硬件自动保存寄存器、切换到内核栈、Ring 改为 0
    ↓
内核处理完毕执行 sysret
    ↓
CPU 硬件自动恢复寄存器、切回用户栈、Ring 改为 3
```

类比：进入政府机密区域，门是硬件门禁，不是靠人站岗，软件绕不过去。

### C 程序编译成机器码的过程

```
源代码 (.c)
    ↓ 预处理（展开 #include、#define）
    ↓ 编译（C 代码 → 汇编代码）
    ↓ 汇编（汇编代码 → 二进制目标文件 .o）
    ↓ 链接（合并多个 .o 和库函数）
可执行文件（机器码）
```

观察中间产物：
```bash
gcc -E file.c -o file.i   # 只预处理
gcc -S file.c -o file.s   # 只编译到汇编
gcc -c file.c -o file.o   # 只编译到目标文件
gcc -o file file.c        # 完整编译
```

编译器本身（gcc）也是机器码，是用已有的编译器编译自己得到的，这个过程叫**自举（bootstrapping）**。你 MacBook 上所有运行的东西（内核、shell、应用）本质都是机器码，苹果预先编译好装进去的。

### errno 是什么

每次系统调用失败，内核把错误原因写进 `errno`（多线程下是线程本地的）。用 `strerror(errno)` 把错误码转成可读字符串：

```c
if (open("file.txt", O_RDONLY) < 0)
    fprintf(stderr, "error: %s\n", strerror(errno));
    // 输出：error: No such file or directory
```

`errno` 是"可修改的整型左值"，意思是它可以像普通变量一样被读写，但底层可能是宏实现（多线程安全需要）。

### fork() 的返回值

```c
pid_t pid = fork();

if (pid < 0) {
    // 出错
} else if (pid == 0) {
    // 子进程（返回值是 0）
} else {
    // 父进程（返回值是子进程的 PID）
}
```

fork 之后父子进程代码、数据完全一样，但各自独立运行，修改一个不影响另一个。子进程执行 `execlp` 成功后整个进程被新程序替换，后面的代码不复存在。

`execlp(buf, buf, (char *)0)` 三个参数：程序名、argv[0]（惯例是程序名本身）、参数列表结束标记。要带参数用 `execvp` 传数组：
```c
char *args[] = {"ls", "-l", "/tmp", NULL};
execvp("ls", args);
```

### const 在 C 中的含义

```c
void  perror(const char *msg);   // const：承诺不修改指针指向的内容
char *strerror(int errnum);      // int：值传递，传的是拷贝，const 没意义
```

`const` 只对指针参数有意义——函数拿到的是地址，可以通过地址修改原始数据，`const` 是对调用者的承诺"我不会改你的数据"。

基本类型（int、char）传进函数是值的拷贝，函数内部怎么改都影响不了外面的变量，加 `const` 没有意义。

**C 的 `const` vs Python/Java 的不可变：**

| | C 的 const | Python/Java 不可变 |
|--|-----------|-------------------|
| 本质 | 编译器警告，可强行绕过 | 语言层面硬性保证 |
| 能否绕过 | 能（强制类型转换） | 不能 |
| 作用对象 | 指针访问权限 | 对象本身的属性 |

```c
// C：const 可以强行绕过
void foo(const char *s) {
    char *p = (char *)s;  // 去掉 const
    p[0] = 'X';           // 编译器不报错
}
```

```python
# Python：不可变是语言保证，绕不过去
s = "hello"
s[0] = 'X'  # 直接报错 TypeError
```

C 给你最大的自由，但出了问题自己负责。高级语言做了更多限制和保护，代价是少了一些灵活性。

### 32位 vs 64位系统

核心差别是 CPU 一次能处理的数据宽度：

| | 32位 | 64位 |
|--|------|------|
| 寄存器宽度 | 32位 | 64位 |
| 最大内存寻址 | 2^32 = 4GB | 2^64 = 理论16EB |
| 指针大小 | 4字节 | 8字节 |
| time_t | int32_t，2038年溢出 | int64_t，2920亿年后溢出 |

物理存储上 64位只比 32位多占一倍空间（4字节→8字节），但能表示的数值范围是 **43亿倍**，因为每多一位信息量翻倍（指数增长）。

**为什么不直接从16位跳到64位：**

够用就好，过度设计有成本。当年 32 位的 4GB 上限看起来"永远够用"，直接跳到 64 位意味着：
- CPU 设计更复杂，成本更高
- 每个指针从 4 字节变 8 字节，内存占用翻倍
- 所有软件要重写

没有人能准确预见几十年后的需求。这是工程领域的普遍规律：**够用就好，真到瓶颈了再升级**。就像写代码不会因为"万一以后有10亿用户"就一开始设计分布式架构。
